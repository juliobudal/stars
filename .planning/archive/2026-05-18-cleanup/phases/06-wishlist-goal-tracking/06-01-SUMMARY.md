## Decisions Made

- **One callback with combined lambda, not two callbacks** — `if: -> { saved_change_to_points? || saved_change_to_wishlist_reward_id? }` instead of two `after_update_commit` lines. Single source of broadcast work for the wishlist card.
- **Bare `have_broadcasted_to("kid_<id>")` matcher (no `.from_channel(Turbo::StreamsChannel)` chain)** — mirrors the working pattern at `spec/services/rewards/redeem_service_spec.rb:35`. The chained form returned 0 broadcasts in initial run; bare form correctly captures `Turbo::StreamsChannel.broadcast_replace_to` invocations.
- **Created the placeholder `_goal.html.erb` partial in this plan instead of deferring to Plan 06-03** — see Deviations §1 below.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Created placeholder kid/wishlist/_goal.html.erb in Plan 06-01**

- **Found during:** Task 3 (running `make rspec ARGS=spec/models/profile_spec.rb`)
- **Issue:** The plan's Critical Rules section asserted that `broadcast_replace_to` would still trigger the broadcast even with a missing template ("logs a missing-template warning but won't crash"). In practice `Turbo::StreamsChannel.broadcast_replace_to` renders the partial **synchronously** before broadcasting; when the partial is missing, `ActionView::MissingTemplate` raises BEFORE `ActionCable.server.broadcast` is called. The `rescue StandardError` in `broadcast_wishlist_card` swallows the exception, so the broadcast assertion (`have_broadcasted_to "kid_<id>"`) sees 0 broadcasts and fails.
- **Fix:** Created a minimal `app/views/kid/wishlist/_goal.html.erb` rendering just the `turbo_frame_tag dom_id(profile, :wishlist)` shell. Plan 06-03 (Wave 1) will overwrite it with the real `Ui::WishlistGoal::Component` render.
- **Files modified:** `app/views/kid/wishlist/_goal.html.erb` (created)
- **Verification:** Broadcast specs pass (`have_broadcasted_to` captures the broadcast).
- **Committed in:** `859a133` (Task 3 commit)

**2. [Rule 1 - Bug] Removed `.from_channel(Turbo::StreamsChannel)` from spec broadcast assertions**

- **Found during:** Task 3 (initial test run)
- **Issue:** Plan's spec template chained `.from_channel(Turbo::StreamsChannel)` on the `have_broadcasted_to` matcher. This filter expected broadcasts to originate from a `Turbo::StreamsChannel` ActionCable channel instance, but the broadcast helper uses class-level `Turbo::StreamsChannel.broadcast_replace_to` which writes directly to `ActionCable.server.broadcast(broadcasting_for(stream), html)` without instantiating the channel — so the matcher saw 0 matching broadcasts.
- **Fix:** Dropped `.from_channel(Turbo::StreamsChannel)` from all three broadcast assertions, mirroring the working pattern at `spec/services/rewards/redeem_service_spec.rb:35` which also broadcasts via `Turbo::StreamsChannel.broadcast_*_to` and asserts with the bare matcher.
- **Files modified:** `spec/models/profile_spec.rb`
- **Verification:** Broadcast specs pass.
- **Committed in:** `859a133` (Task 3 commit)
