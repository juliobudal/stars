  <name>Task 1: Create app/javascript/pwa.js</name>
  <read_first>
    - app/javascript/application.js (current import shape)
  </read_first>
  <behavior>
    - File loaded on every page (via application.js import chain)
    - Registers SW once per page load; logs `[pwa] sw registered, scope: /` to console
    - On `updatefound`: tracks the installing worker; on its `statechange === "installed"` AND `navigator.serviceWorker.controller` truthy, dispatches `pwa:update-available` on window with the registration as detail
    - Gracefully no-ops in browsers without serviceWorker support (Safari old)
  </behavior>
  <action>
    Create `app/javascript/pwa.js`:

    ```js
    // PWA service worker registration + update detection.
    // SW lifecycle background: web.dev/articles/service-worker-lifecycle

    if ("serviceWorker" in navigator) {
      window.addEventListener("load", () => {
        navigator.serviceWorker
          .register("/service-worker", { scope: "/" })
          .then((registration) => {
            console.info("[pwa] sw registered, scope:", registration.scope);

            registration.addEventListener("updatefound", () => {
              const installing = registration.installing;
              if (!installing) return;

              installing.addEventListener("statechange", () => {
                if (installing.state === "installed" && navigator.serviceWorker.controller) {
                  // A new SW has installed and is waiting; an old one still controls the page.
                  window.dispatchEvent(
                    new CustomEvent("pwa:update-available", { detail: { registration } })
                  );
                }
              });
            });
          })
          .catch((err) => {
            console.warn("[pwa] sw registration failed:", err);
          });
      });
    }

    // Optional: when the controller changes (after user-accepted reload), reload page once.
    let refreshing = false;
    if ("serviceWorker" in navigator) {
      navigator.serviceWorker.addEventListener("controllerchange", () => {
        if (refreshing) return;
        refreshing = true;
        location.reload();
      });
    }
    ```

    Notes:
    - Register at `load` event, not `DOMContentLoaded`, so SW download doesn't compete with initial render.
    - `register("/service-worker")` matches the named route from Plan 07-01 (no `.js` extension — Rails route serves it).
    - `controllerchange` reload is the standard recipe to apply a freshly activated SW after user accepts the toast.
  </action>

<task type="auto" tdd="false">
  <name>Task 3: Create pwa_update_controller.js Stimulus controller</name>
  <read_first>
    - app/assets/controllers/fx_controller.js (Stimulus shape)
    - app/components/ui/flash/component.html.erb (toast styling pattern)
    - DESIGN.md §6 Components (existing toast/flash patterns)
  </read_first>
  <behavior>
    - When `pwa:update-available` fires on window, the controller's host element becomes visible (removes `hidden` attribute) and shows two buttons: "Atualizar" and "Depois"
    - "Atualizar" calls `registration.waiting.postMessage({ type: "SKIP_WAITING" })` then the controllerchange listener in pwa.js reloads
    - "Depois" sets `sessionStorage["pwa-update-dismissed"] = "1"` and hides the host
    - On controller `connect`: if `sessionStorage["pwa-update-dismissed"] === "1"`, never show even if event fires this session
  </behavior>
  <action>
    Create `app/assets/controllers/pwa_update_controller.js`:

    ```js
    import { Controller } from "@hotwired/stimulus"

    export default class extends Controller {
      static targets = ["banner"]

      connect() {
        this.dismissed = sessionStorage.getItem("pwa-update-dismissed") === "1"
        this.registration = null
        this._onAvailable = (e) => this.show(e.detail.registration)
        window.addEventListener("pwa:update-available", this._onAvailable)
      }

      disconnect() {
        window.removeEventListener("pwa:update-available", this._onAvailable)
      }

      show(registration) {
        if (this.dismissed) return
        this.registration = registration
        this.element.hidden = false
      }

      apply() {
        const waiting = this.registration && this.registration.waiting
        if (waiting) {
          waiting.postMessage({ type: "SKIP_WAITING" })
          // pwa.js controllerchange listener handles location.reload()
        } else {
          location.reload()
        }
      }

      dismiss() {
        sessionStorage.setItem("pwa-update-dismissed", "1")
        this.dismissed = true
        this.element.hidden = true
      }
    }
    ```

    Also: add a `message` listener inside `service-worker.js` (Plan 07-03 file) so the SW responds to `SKIP_WAITING`:

    Edit `app/views/pwa/service-worker.js` to add this BEFORE the `fetch` listener:

    ```js
    self.addEventListener("message", (event) => {
      if (event.data && event.data.type === "SKIP_WAITING") {
        self.skipWaiting();
      }
    });
    ```

    NOTE: Plan 07-03 already has `self.skipWaiting()` in the install handler. The message handler here is for the case where a NEW worker is in `waiting` state (an old client still controls) — message-based skipWaiting is the standard way to upgrade users without the install-time skipWaiting.

    Decision: actually since Plan 07-03 already calls `skipWaiting()` on install, the new SW will activate immediately. The message handler is belt-and-suspenders for the case where future versions of Plan 07-03 want to drop install-time skipWaiting. Keep it for forward-compat.
  </action>

<threat_model>
| Threat | Disposition | Mitigation |
|--------|-------------|------------|
| Toast persists across sessions, irritates users | mitigate | sessionStorage dismissal (resets each tab close) |
| User stuck on old SW version | mitigate | `controllerchange` reload + manual "Atualizar" CTA |
| SKIP_WAITING postMessage to wrong scope | n/a | postMessage targets the registration's own waiting worker only |
| pwa.js loads on pages with no consent (login) | accept | SW registration only — no PII; safe pre-auth |
</threat_model>
