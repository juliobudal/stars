<objective>
Add the `wishlist_reward_id` foreign key to `profiles` with `on_delete: :nullify`, wire the
`belongs_to :wishlist_reward` association on `Profile`, and add a single `after_update_commit`
broadcast callback so any change to `points` OR `wishlist_reward_id` re-renders the wishlist
Turbo Frame on the kid's per-profile channel. This is the foundation every other plan depends on.

Purpose: Single source of truth for wishlist state + single broadcast source (model callback,
not service) so Tasks::ApproveService and Rewards::RedeemService stay untouched on broadcast
duty. Eliminates double-broadcast risk identified in RESEARCH.md Q2 / A6.

Output: Migration + Profile model changes + extended profile_spec.rb covering association,
nullify-on-delete, and broadcast firing on points and wishlist_reward_id changes.
</objective>

        it "broadcasts to kid_<id> when points change" do
